Rust review: Closing thoughts

June 26, 2018 · About 3 minutes

This article is part number 10 of the Rust review series.

This post is basically a miscellaneous list of good and bad things about Rust that I jotted down while reading the book. I will elaborate on some of these topics in more detail in subsequent posts, but I felt that keeping a high-level itemization here would be useful. I will follow the same good/bad/ugly structure I used back then for the Go post, but this is not as exhaustive as it should be.

For a 3-year old language, it’s pretty good.

The neutral

  • Rust’s default coding style is something that aligns closely with my personal preferences. Doesn’t matter at all when using a language, but I’ve always preferred 4-space indents and snake_case over smaller/bigger indents and CamelCase. Rust matches those well.

The good

  • No implicit type promotions, not even to booleans in conditionals. let foo = 0; if !foo { ... } is just invalid because a negated integer is not a boolean. Pet peeve of mine while reading C code and where everyone likes to treat integers and even pointers as boolean.

  • Lack of a null value. This ensures that all objects are always valid. There are cases when you need objects to exist conditionally, in which case you wrap them with an Option type (which exists in plenty other languages, including Java, Haskell, and in C++’s Boost), but the language will not let you express accesses to those objects without a previous check for their validity: in which case the cost of the check, and the possibility of “nullness” becomes obvious and can be assessed.

  • Generics! OMG, they can exist without wreaking havoc! (C.f. Go.) They work as you may expect, just fine.

  • Imports must be explicitly specified for any symbol external to the current file, which gives clarity as to where they come from. This is useful to understand the dependencies a piece of code has.

The bad

Nothing to say here.

The ugly

  • Traits must be imported explicitly, but there is no direct connection from their import to their use. In other words: there is no visual clue as to what comes from where. At least the compiler warns about unused imported traits, which doesn’t make this a big deal, but it kinda goes against my desire that any symbol used from another file should be explicitly declared and can be syntactically connected from its use to the import.

  • The syntax is cryptic to any newbie, especially lifetime annotations like <'a> and the ? error handling operator. Easy to get used to these, but any newcomer will feel the language is alien.

  • Shadowing of variables feels dangerous as it obscures the principle of single assignment. In practice, this has proven to be necessary when writing certain kinds of constructs to keep the code clearer than it would otherwise be.

  • There is some weird one-off syntax like move or the Sized trait that feel like after-thoughts. They do their job well though, so not complains.

  • Did you like sendmail.cf? Because Rust macros is how you get sendmail.cf again. OMG what an awful thing. Fortunately it seems that macros are still subject to change, so here is hope that they’ll evolve into a more manageable animal. Won’t cover anything else on this subject as I haven’t had the fortune of writing a macro.

  • Strings. Ah, strings, they are a difficult topic. Rust has chosen to have many different string variants and reference types. The common String and &str will suffice for most uses… but as soon as you try to interact with the underlying operating system, it becomes messier because those types enforce UTF-8, while the underlying system may not care.
